# Course Planner AI Planning Session Redesign

Date: 2026-06-28

## Goal

Rebuild the Course Planner first screen as an AI-assisted planning session.

The user should start with a high-level planning intent such as `室内家庭篇` or `厨房`. The system generates structured scene-first chapter candidates, the user accepts, rejects, reorders, and adds revision notes, then the AI revises with the full planning context. Formal chapters are created only after the user locks the selected chapter sequence.

This replaces the current CRUD-style scene category form. The form exposed backend fields (`Category ID`, `Target Level`, `Chapter Count`) and created a dead first-run state when no course existed. Those fields do not belong in the planning UI.

## Product Principles

- Scene-first planning is the primary workflow. The system plans visual memory scenes: spaces, events, object relationships, character actions, and memorable image hooks.
- The user provides intent and feedback, not database fields or manual object lists.
- AI suggestions are candidates. User selection is authoritative.
- Accepted chapter candidates are not overwritten by later AI revisions unless the user explicitly requests a full replan.
- Chapter count is derived from the selected sequence. It is never manually typed.
- IDs are generated by the system. They are never typed by the user.
- Target language and language difficulty are outside this pre-task stage.
- The visual system should follow the existing Pipeline blue-black product theme while using the supplied reference image as layout and density guidance, not as a pixel-perfect source.

## Information Architecture

The first screen is a planning session workspace:

```text
┌──────────────────────┬──────────────────────────────┬──────────────────────┐
│ Planning Brief        │ AI Candidate Chapters         │ Selected Sequence    │
│                      │                              │                      │
│ Intent               │ Candidate cards               │ Accepted chapters    │
│ Revision note        │ Accept / Reject / Revise      │ Reorder / Remove     │
│ Generate / Revise    │ Scene-first details           │ Lock sequence        │
└──────────────────────┴──────────────────────────────┴──────────────────────┘
```

### Planning Brief

Purpose: capture the user's high-level intent and feedback for the current AI planning session.

Controls:

- `规划目标`: a natural-language text area. Examples: `室内家庭篇`, `厨房`, `校园走廊`.
- `修改意见`: a natural-language text area for revising the current candidate set.
- `生成候选方案`: first generation action.
- `根据反馈重生成`: revision action after candidates exist.

The brief does not ask for object lists, events, style notes, IDs, target levels, target languages, or chapter counts.

### AI Candidate Chapters

Purpose: show AI-generated chapter candidates that are not yet formal chapters.

Each candidate card shows:

- `title_zh`: candidate title.
- `scene_brief_zh`: what the scene looks like.
- `event_brief_zh`: what is happening.
- `space_objects_zh`: how key objects participate in the space and event.
- `character_action_zh`: what characters are doing.
- `memory_hook_zh`: the memorable visual hook.
- `design_reason_zh`: why the candidate is worth considering.

Actions:

- Accept: add to selected sequence.
- Reject: mark as rejected for future AI context.
- Revise: revise that candidate using a focused note.

### Selected Sequence

Purpose: make the user's accepted chapter structure explicit.

Behavior:

- Accepted candidates appear in user-controlled order.
- The visible count is derived from selected items.
- Users can reorder or remove accepted candidates.
- `Lock Chapter Sequence` converts selected candidates into formal `Chapter[]`.

The AI must not reorder or replace this list during normal revision. User control is the source of truth.

## Backend Data Model

Add a minimal planning-session boundary instead of overloading formal `Chapter[]` as candidates.

```text
PlanningSession
- id
- intent_text
- status: draft | generated | locked
- planning_summary_zh
- candidates[]
- selected_candidate_ids[]
- rejected_candidate_ids[]
- revision_note
- created_at
- updated_at
```

```text
ChapterCandidate
- id
- title_zh
- scene_brief_zh
- event_brief_zh
- space_objects_zh
- character_action_zh
- memory_hook_zh
- design_reason_zh
```

Formal `Chapter[]` is produced only by locking selected candidates.

`CourseProject` and `Space` should remain backend containers for now. The UI does not expose their internal fields. If no course/space exists for a new planning session, the system creates the minimal container required to persist and later lock the session.

## API Design

Add these endpoints under `/api/course-planner`:

```text
POST /planning-sessions
```

Create a planning session from `intent_text`. The server creates any required internal course/space container.

```text
POST /planning-sessions/{session_id}/ai/candidates
```

Generate or revise candidates. The request includes the current session state and optional `revision_note`. The response returns a full updated session.

```text
PATCH /planning-sessions/{session_id}/selection
```

Update selected/rejected candidate IDs and ordering. This does not call AI.

```text
POST /planning-sessions/{session_id}/lock
```

Convert the selected sequence into formal chapters. The response returns the locked session and the created `Chapter[]`.

## AI Planning Contract

The AI task is to generate or revise scene-first chapter candidates for the user's planning intent.

Prompt inputs:

- Planning intent.
- Existing candidates.
- Accepted candidate IDs in order.
- Rejected candidate IDs.
- Current revision note.
- Focused candidate ID when revising one card.

Prompt objective:

- Produce daily-life scene chapter candidates.
- Make each candidate concrete enough for later scene design.
- Ensure candidates differ in space, event, object relationships, character action, or memory hook.
- Preserve accepted candidates during normal revisions.
- Replace or improve rejected and unselected candidates according to the user's note.

Structured AI output:

```json
{
  "planning_summary_zh": "本轮规划摘要",
  "candidates": [
    {
      "title_zh": "厨房早餐打翻牛奶",
      "scene_brief_zh": "画面描述",
      "event_brief_zh": "事件描述",
      "space_objects_zh": "物品与空间关系",
      "character_action_zh": "角色动作",
      "memory_hook_zh": "视觉记忆点",
      "design_reason_zh": "保留理由"
    }
  ]
}
```

## Frontend Design

The Course Planner route `/course-planner` renders the planning session workspace.

Empty first-run state:

- Show the Planning Brief panel.
- Candidate panel shows an empty state waiting for candidate generation.
- Selected panel explains that accepted candidates will appear there.
- No hidden prerequisite blocks the first action.

Generated state:

- Candidate cards occupy the center.
- Selected sequence occupies the right panel.
- Revision note stays visible on the left so multi-turn feedback is always one action away.

Locked state:

- Selected sequence shows locked chapters.
- Each chapter has an entry point into the chapter scene designer route.

Visual direction:

- Use the Pipeline blue-black palette from the existing app shell.
- Use blue primary actions and restrained status colors.
- Use dense panels and list cards similar to the reference image.
- Avoid the current brown/orange/green Course Planner palette.

## Testing Requirements

Backend tests:

- Planning session can be created from intent without user-provided IDs.
- AI candidate generation writes candidates, not formal chapters.
- Selection update stores accepted/rejected IDs and order.
- Revision request passes existing candidates, selected IDs, rejected IDs, and revision note to the AI provider.
- Lock converts selected candidates into ordered formal chapters.

Frontend tests:

- Empty `/course-planner` first run shows only planning intent workflow and no internal fields.
- `Category ID`, `Target Level`, and `Chapter Count` are absent.
- Generating candidates renders candidate cards with scene, event, object relationship, action, memory hook, and reason.
- Accepting candidates updates the selected sequence count.
- Rejecting candidates sends rejected IDs in the next revision request.
- Revision request includes the full planning context.
- Locking creates chapter designer entry points.

Regression checks:

- Course Planner route still appears in product navigation.
- Chapter workspace still opens for locked chapters.
- Existing Pipeline route remains unaffected.
