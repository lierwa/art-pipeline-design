# Model-Backed Asset Pipeline Redesign

Date: 2026-06-17

## Goal

Rebuild the workbench around a real asset pipeline instead of a demo-style control surface. The first usable version must let a user upload a scene, run model-backed object detection, review and correct boxes, split large detected objects into child assets, merge over-split boxes, and export reviewed assets.

The app must never present weak CV heuristics as intelligent detection. If the configured model provider is unavailable or fails, detection fails with a clear error and no fake proposals are generated.

## Product Shape

The main screen keeps one complete workbench layout:

- A left workflow rail with Upload, Detect, Review, Segment, and Export.
- A central canvas that dominates the page.
- A right asset tree and contextual editor.
- A compact bottom model/status strip.

The UI should expose only the actions that are useful for the current selection state:

- No selection: run detection or upload source.
- Single candidate selected: accept, reject, edit box, add child, run detect inside, split parent.
- Multiple candidates selected: merge, reject selected, clear selection.
- Box editing: apply, cancel, reset to model box.

Advanced geometry fields, provider diagnostics, and export details are available only through compact secondary panels. They are not part of the default visual load.

## Detection Pipeline

The detection pipeline is provider-based:

```text
source image
-> detection provider returns labels, boxes, confidence, provider metadata
-> optional segmentation provider creates masks from accepted/reviewed boxes
-> user reviews and edits proposals
-> accepted assets export to manifest and image/mask outputs
```

Initial provider target:

- Detection: GroundingDINO-style open-vocabulary detector.
- Segmentation: SAM2-style promptable segmentation using approved boxes.

The initial asset vocabulary is configurable and starts with:

```text
cat, bathtub, sink, bathroom cabinet, mirror, window, curtain,
towel, basket, stool, bottle, plant, shelf, rug
```

The system filters provider output before showing it:

- Drop labels outside the configured asset vocabulary.
- Drop generic scene labels such as bathroom, room, wall, floor, object, furniture, background.
- Drop combination labels unless explicitly configured.
- Apply confidence thresholds per label group.
- Apply non-maximum suppression to reduce duplicate boxes.

Detection output records must include:

- `id`
- `label`
- `confidence`
- `bbox`
- `sourceProvider`
- `sourcePrompt`
- `status`
- `parentId`
- `history`

## Candidate Statuses

Candidate statuses should describe workflow, not implementation trivia:

- `model_detected`: raw provider proposal after filtering.
- `edited`: user adjusted label or geometry.
- `child`: user-created or locally detected child of a parent object.
- `merged`: generated by merging multiple selected candidates.
- `accepted`: approved for segmentation/export.
- `rejected`: hidden from normal review but recoverable.
- `exported`: included in the current export manifest.

Rejected or merged-away candidates are not deleted by default. They are hidden from normal review and remain recoverable from history.

## Box Editing

When a candidate is selected, the canvas supports direct box editing:

- Drag the selected box to move it.
- Drag side and corner handles to resize it.
- Use arrow keys for 1px nudge.
- Use Shift plus arrow keys for 10px nudge.
- Reset to model box restores the original provider geometry.

Editing changes the candidate status to `edited` and appends a history entry containing the previous geometry and timestamp.

The right editor shows:

- Label
- Confidence
- Status
- Parent relationship if present
- Primary actions

Raw bbox numeric fields are collapsed under Advanced.

## Split Into Child Assets

When a large detected object contains smaller exportable assets, the user can split it without losing the parent.

Example:

```text
cabinet
  plant
  pink bottle
  white bottle
  towel roll
  cat ornament
```

Interaction:

1. User selects the parent candidate.
2. User chooses Add child, Split parent, or Run detect inside.
3. The canvas focuses on the parent region.
4. User either draws child boxes manually or runs local detection inside the parent box with a narrowed vocabulary.
5. New child candidates are created with `parentId` set to the parent candidate.
6. The parent remains visible as a container candidate.

Default export behavior:

- Child candidates can be accepted and exported.
- Parent candidates are not exported if accepted children exist, unless the user explicitly marks the parent for export.

## Merge Over-Split Boxes

When a model splits one intended asset into several boxes, the user can merge them.

Interaction:

1. User selects multiple candidates with Shift-click, checkboxes, or lasso selection.
2. The right panel switches to a multi-selection action bar.
3. User chooses Merge into one asset.
4. The app previews the merged bbox as the union of selected boxes.
5. User confirms or edits the merged label.
6. A new candidate is created with status `merged`.
7. Original candidates are marked as merged into the new candidate and hidden from the default review list.

Segmentation should run from the merged bbox rather than blindly stitching the old masks.

## Asset Tree

The right panel uses a compact tree to show structure:

- Parent candidates can expand and collapse.
- Children are indented under parents.
- Status chips show `model`, `edited`, `child`, `merged`, `accepted`, or `rejected`.
- Selecting a tree row highlights the canvas box.
- Selecting a canvas box selects the matching tree row.

The tree solves two problems:

- Users can understand parent/child splits.
- Users can recover hidden or merged candidates without digging through raw data.

## Canvas Behavior

The canvas is the primary workspace.

It supports:

- Pan and zoom.
- Box overlays with readable labels.
- Selected box handles.
- Parent/child visual distinction.
- Multi-select visual state.
- Merge preview outline.
- Optional mask overlay after segmentation exists.

The default view should not show every diagnostic overlay at once. It should prioritize accepted, selected, and needs-review candidates.

## Export

V1 export includes only reviewed assets:

- Accepted standalone candidates.
- Accepted child candidates.
- Accepted merged candidates.
- Explicitly accepted parent candidates when the user chooses to export the parent.

Export writes:

- Asset manifest
- Source image metadata
- Candidate label and status
- Bbox
- Parent relationship
- Provider metadata
- Mask path when segmentation has run

Default export blocks accepted assets that do not have a generated mask. A separate bbox-only debug export mode may be added for inspection, but it must be visibly labeled as debug output and must not be the default asset-pack export path.

## Errors And Empty States

Important failures must be visible and actionable:

- No provider configured: show provider setup error.
- Provider failed: show provider error and retain previous reviewed state.
- No detections found: show empty detection result and suggest vocabulary/threshold changes.
- Segmentation failed: leave boxes intact and mark masks as missing.

The app must not silently fall back to same-color components, edge components, or any heuristic that produces fake model proposals.

## Testing Strategy

Tests must verify user-value behavior, not self-fulfilling implementation details.

Required test coverage:

- Detection fails clearly when no provider is configured.
- Detection provider output is normalized into candidates with label, confidence, bbox, provider metadata, and status.
- Generic labels and out-of-vocabulary labels are filtered.
- Duplicate boxes are reduced by non-maximum suppression.
- Editing a box changes geometry, preserves original geometry in history, and marks status as edited.
- Splitting a parent creates child candidates with parentId and keeps the parent recoverable.
- Merging selected candidates creates a new candidate and hides originals as merged history.
- Export includes accepted standalone, child, and merged assets only.
- UI state shows only relevant actions for no selection, single selection, multi-selection, and edit mode.

Fixture tests must use meaningful expectations against representative scene images. They must not assert only that "some boxes exist."

## Non-Goals For V1

The redesign should not include these until the core loop works:

- Codex repair packaging
- Inpainting/completion workflows
- Complex QA dashboards
- Freehand mask painting as a primary path
- Multi-scene project management
- Marketplace/provider selection UI

These can be added after upload, detect, review, split, merge, segment, and export are trustworthy.
